Phases tab — what you're looking at

Phases are how the kit organizes and tracks your project. Each phase is a collapsible section added automatically when work begins, and each contains a set of tasks that Claude creates, estimates, and executes — complete with the step-by-step instructions it follows. The table below shows Baseline (senior dev, no AI) and With AI hour estimates for every task, with actuals filled in as each step is logged via /log-time.

  1. Phase 0 just ran. init-project.sh performed 15 bootstrap steps (file copies, scaffolding, hooks install) and logged wall-clock actuals to .claude/time-log.json. Open this dashboard to see them below. Click the glowing checklist icon next to the Phases tab to load more robust sample data from a real project and see examples of how the kit was used to build itself.
  2. Baseline vs With AI Estimate Columns. Baseline = a senior dev working alone. With AI = same dev with an AI pair-programmer. The gap between these two columns is the kit's built-in efficiency case study — it grows phase by phase as real actuals come in.
  3. Green / red on Difference means the step finished under (green) or over (red) the With AI estimate. New project: every cell shows — until actuals are logged via /log-time.
  4. Buffer input. Type a percentage to add contingency to a phase. The buffered-total row and grand total update live. Phase 0 default is 20% — typical for first-time setup variance.
  5. Sample data button (the checklist icon at the left of the tab bar below) — click it to load actuals from a completed project so you can preview exactly what a fully-populated dashboard looks like. Click again to clear.
  6. Refresh (↻) reloads the page, resetting all panels. Expand / collapse all (chevron) opens or closes every phase row at once — or click any phase row header individually.
  7. ⓘ menu → Estimation Rubric shows the three-table rubric (task type, size, unknowns) behind every Baseline and With AI Estimate. Run /estimate in Claude Code to apply it automatically to any new step.

Agents tab — what gets tracked here

If you just completed Phase 0, the list below will be empty — no agents have run in your project yet. Once you complete the Integration Wizard and define your project goals, agents will plan phases, create tasks, and begin executing work. As they do, each spawn is logged here automatically. Click the glowing bot icon next to Agent spawn log to load sample data and preview what a populated agent log looks like.

  1. AGENT-LOG.md is the source of truth for every subagent spawn: Claude built-in agents, VoltAgent specialists, CCPM parallel workers, MCP calls. The AI is the sole writer — never edit it directly.
  2. Automatic logging. The log-agent-on-spawn.sh PreToolUse hook fires before every Agent tool call and writes a row automatically. Use /log-agent to manually log MCP invocations the hook cannot catch.
  3. Framework column classifies spawns as builtin, voltagent, ccpm, mcp, or custom. The KPI tiles break down the total count per framework.
  4. Group column (G1, G2 …) marks agents that ran in parallel within the same fan-out. CCPM parallel workers and VoltAgent review fans appear here as colored group pills.
  5. Sample data button (Glowing bot icon next to Agent spawn log) — click to preview what agent tracking looks like on a multi-phase project with VoltAgent specialists and CCPM parallel workers. Click again to clear.
  6. Refresh (↻) reloads the page. Expand / collapse all (chevron in the tab bar) opens or closes all phase groups in the spawn log. Click the sliders icon within any phase row to filter KPIs to that phase only.

Generate a theme from any URL

dembrandt is a CLI that scrapes any website and emits a DESIGN.md in the same format every theme above uses. Paste a URL, copy the command, and follow the steps to add it to your project.

dembrandt https://example.com --design-md

Then, in your project root:

  1. Run the command above; dembrandt writes output/<slug>/DESIGN.md.
  2. Move it to docs/themes/<slug>/DESIGN.md (or just docs/DESIGN.md if it's the only theme).
  3. Open DASHBOARD.html in an editor; add a body[data-theme="<slug>"] block with the same 18 --color-* vars every existing theme defines (copy any theme above as a template).
  4. Add an entry to the JS THEMES array near the bottom: { slug: '<slug>', name: '<Name>', canvas: 'light|dark', accent: '#…' }.
  5. Reload — your new theme appears in the picker.

A future kit version will automate steps 2-4; for now this stays a manual drop-in so the dashboard remains a single self-contained HTML file.
