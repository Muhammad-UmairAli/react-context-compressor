# Task: Core sanitization + security audit

## Description

Add the client-side data-safety layer: detect and remove/redact sensitive fields
(tokens, credentials, private IDs) before state can leave the client. Composed into
the compression walk from 002. Ships a sensible default deny-list that consumers can
extend or replace. This task is a trust boundary — it gets a dedicated security audit
(R=true).

## Acceptance Criteria

- [ ] `sanitize` matchers accept exact strings and `RegExp`; matched fields are removed
      or redacted to a configurable marker (default `"[REDACTED]"`).
- [ ] A default deny-list ships (e.g. `token`, `password`, `secret`, `apiKey`,
      `authorization`, `creditCard`, `ssn`) and is extendable/replaceable via options.
- [ ] Matching applies at any nesting depth and inside arrays.
- [ ] **No-leak guarantee:** a configured sensitive key never appears (as key or value)
      anywhere in the output — asserted by an automated test across the matrix
      (top-level, nested, in-array, regex match, default-list match).
- [ ] Sanitization runs within the single compression pass (not a second full traversal).
- [ ] Key matching is case-insensitive by default (documented), to avoid `Token` vs
      `token` bypass.

## Technical Details

- Implement in `src/core/sanitize.ts`; integrate into the 002 walker so one pass does
  both. Export sanitize options through `CompressOptions`.
- Be careful that redaction does not accidentally re-expose values via array indices or
  Map entries.
